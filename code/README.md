# Multi-Modal Evidence Review — Solution

This solution implements a robust, multi-threaded evaluation pipeline for processing user claims against image evidence. The final architecture uses a single-shot analyzer approach (`pipeline/analyzer.py`) which evaluates the claim conversation, user history, requirements, and all attached evidence images in exactly ONE API call per row. This unified approach provides superior accuracy and significantly reduces API call costs compared to a sequential extraction-then-vision pipeline.

## Setup
1. Install dependencies:
   ```bash
   pip install -r requirements.txt
   ```
2. Create a `.env` file in the `code/` directory with the following variables:
   ```env
   OPENAI_API_KEY=your_api_key_here
   OPENAI_BASE_URL=https://YOUR_AZURE_ENDPOINT/openai/v1
   ```

## How to run
To process the dataset and generate the final output:
```bash
python main.py
```
This reads `dataset/claims.csv` and writes the final predictions to `output.csv`.

## How to run evaluation
To run the evaluation script against the sample dataset:
```bash
python evaluation/main.py
```
This script evaluates Strategy A vs Strategy B on `dataset/sample_claims.csv` and overwrites `evaluation/evaluation_report.md` with the exact accuracy comparison.

## Caching
Results from the Azure LLM endpoint are cached persistently on disk in the `code/.cache/` directory to speed up reruns and prevent redundant API calls. The cache keys are strictly generated by content hashing. To force a completely fresh run across the pipeline, simply delete the `.cache/` folder before executing.

## Strategy Selection
Strategy A (single-shot analyzer) was selected as the final architecture. An alternative multi-stage extraction/vision pipeline (Strategy B) was initially prototyped but discontinued. Strategy A correctly reasons across the entire context holistically in a single pass, which proved to be substantially cheaper, faster, and more robust than issuing multiple sequential API calls for the same claim.
