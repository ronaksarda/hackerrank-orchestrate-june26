# Multi-Modal Evidence Review — Solution

This solution implements a robust, multi-threaded evaluation pipeline for processing user claims against image evidence. The final architecture uses a single-shot analyzer approach (`pipeline/analyzer.py`) which evaluates the claim conversation, user history, requirements, and all attached evidence images in exactly ONE API call per row. This unified approach provides superior accuracy and significantly reduces API call costs compared to a sequential extraction-then-vision pipeline.

## Setup
1. Install dependencies:
   ```bash
   pip install -r requirements.txt
   ```
2. Create a `.env` file in the `code/` directory with the following variables:
   ```env
   OPENAI_API_KEY=your_api_key_here
   OPENAI_BASE_URL=https://YOUR_AZURE_ENDPOINT/openai/v1
   ```

## How to run
To process the dataset and generate the final output:
```bash
python main.py
```
This reads `dataset/claims.csv` and writes the final predictions to `output.csv`.

## How to run evaluation
To run the evaluation script against the sample dataset:
```bash
python evaluation/main.py
```
This script evaluates the single-shot analyzer strategy on `dataset/sample_claims.csv` and outputs accuracy metrics to the terminal.

## Caching
Results from the Azure LLM endpoint are cached persistently on disk in the `code/.cache/` directory to speed up reruns and prevent redundant API calls. The cache keys are strictly generated by content hashing. To force a completely fresh run across the pipeline, simply delete the `.cache/` folder before executing.

## Strategy Selection
Strategy A (**single-shot analyzer**) was selected as the final architecture. An alternative multi-stage extraction/vision pipeline (Strategy B) was initially prototyped but discontinued. 
- **Context Sharing:** A single-shot approach correctly reasons across the entire context holistically. If an LLM parses text first and a vision model looks at images second, cascading errors occur. Passing everything in one prompt is more robust.
- **Cost Efficiency:** Making 1 API call per row instead of 3 means using 1/3rd of the prompt tokens.
- **Speed:** 1 API call avoids hitting the Azure strict Requests Per Minute limits.

## Adversarial Defense
The pipeline natively handles adversarial behavior (prompt injections, fraudulent history) via:
- **Pre-processing:** Actively sanitizing known jailbreak phrases (e.g., "ignore all previous instructions") from the user's chat input before API submission, cleanly bypassing the Azure `ResponsibleAIPolicyViolation` content filter block.
- **Strict Prompting:** Instructing the model that sticky notes or text instructions visible in an image do not constitute physical damage.

## Rate Limit Handling
To guarantee 100% throughput reliability against strict Azure 429 "Too Many Requests" limiters on the dev-tier, the pipeline implements three layers of defense:
1. **Concurrency Control:** Strict sequential processing via ThreadPoolExecutor (`max_workers=1`).
2. **Pacing:** An intentional `time.sleep(2.0)` stagger between dispatching requests.
3. **Exponential Backoff:** Tenacity `@retry` decorator wrapping the LLM call with a 1-30s random exponential backoff.
